# PCI DSS 培训演讲词

## 01. PCI DSS 培训要点清单

今天不是给大家念合规条文，而是把 PCI DSS 翻译成产品、架构、运营三个人都能听懂的语言。

这场培训走四个模块：合规判断、产品落地、技术架构、运营审计。走完以后，大家应该能回答三件事：第一，我们公司到底在不在 PCI Scope 里；第二，我们的产品选型有没有把 PCI 成本压到最小；第三，如果真的出事，我们怎么兜底。

PCI DSS 不应该只被看成安全团队或者合规团队的事情。它本质上会影响支付产品怎么设计、系统边界怎么划、数据怎么流、团队日常怎么运营。

## 02. 目录 · 四大模块

今天的结构很简单，四个模块按决策顺序展开。

第一是合规判断，先看等级、SAQ 和 Scope。这个阶段回答的是：要求是什么，我们到底承担多大责任。

第二是产品落地，讲 Hosted Page、Elements、Direct API、CIT/MIT、Token 和多 PSP。这个阶段回答的是：怎么通过产品形态把合规范围降下来。

第三是技术架构，讲 Vault、密钥、前端支付页安全和日志脱敏。这个阶段回答的是：如果业务确实要碰卡数据，工程上怎么把风险关住。

第四是运营审计，讲客服、测试、权限、事件响应。这个阶段回答的是：怎么持续证明我们做到了，而不是年底突击补材料。

PCI DSS 对很多人来说是“年底突击一次”的事，但实际上它贯穿产品从 0 到 1 的每一个决策。选 Hosted Page 还是 Direct API，Scope 可能差十倍；要不要自建 Vault，合规成本可能差到百万级。

## 03. 模块一：合规判断

模块一先解决三个问题：PCI DSS 到底是什么，你属于哪一级，你的 Scope 有多大。

这三个问题必须先答对。等级判断错了，验证方式会错；SAQ 判断错了，审计路径会错；Scope 判断错了，后面架构隔离、权限控制、日志治理都会跟着错。

所以这一部分不讲细碎条文，重点讲判断逻辑。

## 04. 补充：PCI 与支付牌照的定位

这一页先澄清一个很常见的误解：PCI DSS 和支付牌照不是一回事。

PCI DSS 是卡组网络的安全准入要求。只要你处理、存储或传输持卡人数据，就必须满足 PCI DSS。它回答的是：你有没有资格安全地接入卡组网络、处理卡数据。

支付牌照是本地监管和经营许可问题。它回答的是：你能不能在某个国家或地区合法开展资金相关业务，尤其是收单、清结算、资金归集这类活动。

所以可以出现几种情况。你不一定需要支付牌照，也能做信用卡收单，比如跨境商户通过收单机构合作完成资金清算。但你只要碰到卡数据，就必须满足 PCI DSS。

反过来，有些本地收单业务、资金清结算业务、支付机构业务，可能需要当地支付牌照或类似许可。这取决于当地法律法规、你是否直接参与资金清算、你在链路中的角色是什么。

这页的关键结论是：PCI 是安全底线，支付牌照是经营许可。两者经常同时出现，但不是同一类要求。先把这个边界讲清楚，后面再讲 Level、SAQ、Scope，大家会更容易理解。

## 05. 一、定位与边界

PCI DSS 不是一份泛泛的安全条文，它更像是一张支付责任边界图。它关注的是：谁接触卡数据，卡数据流经哪里，哪些系统因此进入 Scope。

很多团队会把 PCI DSS 当成 ISO27001 那种通用安全清单来抄，这个方向是错的。PCI DSS 只盯一件事：卡数据有没有被你碰到。

举个例子，公司可能有 200 个微服务，但真正进入 Scope 的可能只有支付 API、Vault、数据库、日志系统、跳板机这几个。剩下的服务只要网络隔离和访问控制做好，就不一定进入 PCI Scope。

产品同学要特别注意第二点：PCI 的成本通常不是审计那天决定的，而是产品形态定下来那一刻决定的。Hosted Page、Elements、Direct API，看起来只是支付体验差异，背后其实是完全不同的合规成本。

v4.0 的核心变化是从“年度大考”变成“持续运营”。以前很多团队 Q4 突击三个月还能过，现在要求你持续监控、持续评估，日志告警、变更管理、脚本监控都要能证明持续有效。

## 06. 二、认证等级与验证方式

认证等级先分两类身份：Merchant 和 Service Provider。

如果你是为了自己业务收款，一般是 Merchant。如果你帮别人处理支付，尤其是平台、支付网关、Vault 服务商，就可能是 Service Provider。身份不同，验证要求不同。

Merchant Level 1 到 Level 4 通常按年交易量划分，Level 1 是最高等级，门槛通常是年交易量超过 600 万笔。Service Provider 一般分 Level 1 和 Level 2，Level 1 往往需要 ROC 和 QSA 现场审。

这里最容易混淆的是 SAQ 类型。SAQ 不是安全团队拍脑袋选的，而是接入方式决定的。Hosted Payment Page 通常走 SAQ A；iframe 或 Elements 走 SAQ A-EP；Direct API 基本会走 SAQ D。

所以产品形态本身就是合规卖点。如果你能帮商户从 SAQ D 降到 SAQ A-EP，他们的合规工作量可能少 80%。这不是安全团队的事情，而是产品竞争力。

验证材料包括 SAQ、AOC、ROC、ASV 扫描和渗透测试。v4.0 之后还要能证明控制不是某一天有效，而是持续有效。

## 07. 补充：SAQ 决策树

这一页把 SAQ 判断讲成一棵决策树。

第一步先问：卡数据是否进入商户系统。如果卡号完全不进入商户系统，用户是在 PSP 或收单机构托管页面输入卡信息，商户通常可以走 SAQ A。

如果卡号不进入商户后端，但支付页由商户页面承载，比如 iframe 或 Elements，商户前端仍然要承担支付页安全责任，通常会落到 SAQ A-EP。

如果卡号直接进入商户前端或后端，也就是 Direct API 模式，商户基本就进入完整 CDE 控制范围，通常走 SAQ D。

这一页的关键结论是：SAQ 不是安全团队想选哪个就选哪个，而是产品接入形态的结果。产品设计越早考虑 PCI，后面的合规成本越可控。

## 08. 三、PCI Scope

Scope 是 PCI DSS 里最容易被低估的部分。很多人以为“我的服务器没有直接收卡号，所以我不在 Scope 里”，这个判断经常不成立。

进 Scope 的不只是支付页。API、Vault、数据库、缓存、消息队列、日志、APM、BI、客服后台、CI/CD、跳板机，都可能因为接触、传输、存储或影响卡数据环境而进入 Scope。

常见盲区有几个。第一，错误日志里打出了完整 PAN；第二，APM 采集了请求体；第三，客服截图里有卡号；第四，测试环境用了真实卡；第五，风控或 BI 同步了未脱敏数据。

只要 ELK 里出现一条完整 PAN，整个 ELK 集群就可能被拉进 CDE。APM 采了请求体，APM 也会进 Scope。客服系统能看到完整卡号，客服后台也会进 Scope。

降 Scope 的思路可以记成七件套：Hosted Page、iframe、PSP token 替代 PAN、CDE 隔离、日志脱敏、最小权限、不存 CVV。

更抽象一点说：能不碰就不碰；碰了就脱敏；脱敏不了就隔离；隔离不了就加密。CVV 是铁律，永远不要存，存了就是审计否决项。

## 09. 补充：CDE Scope 边界放大图

这一页是 Scope 页的放大版。

CDE 核心区通常包括支付 API、Vault、PAN 数据库、KMS 或 HSM。它们直接处理、存储或保护卡数据，是最确定会进入 Scope 的部分。

真正容易扩大 Scope 的，是外围系统。比如日志 ELK、APM Trace、客服后台、BI 数仓、MQ、缓存、CI/CD。它们看起来不是支付核心系统，但只要存储、处理、传输卡数据，或者能影响 CDE 安全，就可能被拉进 Scope。

所以判断标准不是系统名字，而是它和卡数据、CDE 安全之间的关系。

架构目标是让 CDE 核心区尽量小。外围系统能脱敏就脱敏，能隔离就隔离，访问权限要最小化，变更路径要受控。这样 PCI 成本才不会从支付系统扩散到整个公司。

## 10. 案例一：一条 PAN 日志把 ELK 拉进 Scope

这个案例用来让大家记住 Scope 的判断标准。

场景是支付 API 报错时，工程为了排查方便，把完整请求体写进 ELK。请求体里包含完整 PAN。

团队的误判是：ELK 只是日志系统，不是支付核心系统，所以不在 PCI Scope 里。

但 QSA 不会按系统名字判断 Scope，而是看数据触点。只要 ELK 里有完整 PAN，ELK、APM、日志访问权限、留存策略、备份和运维账号都可能被纳入 PCI Scope。

正确做法是在入口、日志框架和 APM 三层控制。入口字段级脱敏，日志禁止记录 PAN 和 CVV，APM 关闭请求体采集，并定期做数据发现，确认日志和链路追踪里没有敏感卡数据。

这个案例的结论是：Scope 看数据触点，不看系统名字。

## 11. 模块二：产品落地

模块一讲的是要求是什么，模块二讲的是产品怎么设计，才能把合规成本变成业务收益。

支付产品形态不是单纯的体验问题，它直接决定 PCI Scope。Token 也不是单纯的技术实现，它决定你能不能跨 PSP、能不能做订阅、能不能提升授权率。

这一部分重点给产品和支付业务同学。

## 12. 补充：资金流 / 数据流 / 责任流

这一页先建立一个重要认知：同一笔交易里，资金流、卡数据流、合规责任流不是同一条线。

资金流讲的是钱如何清结算，可能经过 PSP、收单机构、卡组织、发卡行，最后到商户账户。

卡数据流讲的是 PAN、token、授权数据经过哪里。有的模式下卡数据只经过 PSP 或卡组织，有的模式下会进入商户自己的 CDE。

责任流讲的是谁因为接触卡数据、处理卡数据，或者能影响 CDE 安全，而承担 PCI 控制要求。

很多误判来自把这三条线混在一起。比如商户不直接清算资金，不代表不承担 PCI；商户使用 PSP，也不代表所有系统都自动出 Scope。要看卡数据到底流经哪里，以及哪些系统能影响这条链路。

## 13. 四、支付产品形态与 PCI 影响

支付接入形态基本可以分成几类。

Hosted Payment Page 是最省事的模式，用户跳转到 PSP 或收单机构托管的页面输入卡信息。商户基本不碰卡数据，所以 PCI 范围最小，通常可以走 SAQ A。缺点是跳转可能影响转化。

Embedded 或 Elements 是现在很多互联网产品的甜点区。前端嵌入 PSP 提供的 iframe 或组件，后端只拿 token。这个模式体验比跳转好，但前端页面仍然要承担安全责任，通常走 SAQ A-EP。

Direct API 是 Scope 最大的模式。卡号直接进入你的前后端，你就需要承担完整 CDE 的隔离、加密、审计、测试、权限和事件响应。除非你有强理由，比如自建风控、多 PSP 路由、自建 Vault，否则不要轻易碰。

PayFac 是平台模式。平台做 Master Merchant，子商户可以快速 onboarding，但平台要扛 Level 1 级别的 PCI 和运营风险。

Click to Pay 和 Passkey 是近几年的机会。卡数据由卡组织、钱包或设备侧托管，商户尽量不碰卡数据。它们的价值不只是降低 Scope，也包括提升体验和转化。

## 14. 案例二：Direct API 让商户掉到 SAQ D

这个案例讲给产品同学。

场景是产品为了减少一次跳转，把 Hosted Page 改成自建卡输入页，再通过 Direct API 提交到 PSP。表面上看，这是一次支付体验优化。

误判点是只评估了转化率，没有评估 SAQ 和 CDE 成本变化。

后果是商户可能从 SAQ A 或 SAQ A-EP 直接进入 SAQ D。影响不只是多填一份问卷，而是前端支付页安全、后端网络隔离、日志脱敏、客服权限、CI/CD 变更控制、安全测试、事件响应都要按更重的控制要求重做。

正确做法是优先评估 Elements、Click to Pay 或 Hosted Page 的体验优化空间。确实必须 Direct API 时，要先做 CDE 设计、Scope 评估和合规成本评审，再决定是否值得。

这个案例的结论是：支付体验优化不能绕开 PCI 成本评估。

## 15. 补充：Passkey + Click to Pay 现代支付架构

这一页补的是模块二里一个很关键的认知缺口：Passkey 和 Click to Pay 不是同一层能力，但它们组合起来，正好对应现代支付的两个核心问题。

Passkey 解决的是身份认证。它基于 WebAuthn 和 FIDO2，用设备里的生物识别或本地认证能力完成登录。它的价值是减少密码、降低钓鱼风险、降低账号接管风险。简单说，Passkey 让平台更确信“这个人就是这个账户的真实使用者”。

Click to Pay 解决的是支付凭证和支付发起。卡号不是直接进入商户系统，而是通过卡组织网络、发卡行和网络 token 来完成支付。商户侧更多是集成 Click to Pay SDK，发起支付请求，不直接接触完整卡号。

所以这张图要分三层看。第一层是用户设备，用户通过 Passkey 做身份验证，通过 Click to Pay 选择已保存的支付凭证。第二层是商户系统，商户完成登录会话和支付发起，但不存密码、不存卡号、不处理敏感卡数据。第三层是卡组织网络和发卡行，它们负责 token、路由、授权和风控。

这就是为什么它能同时提升体验和降低 PCI Scope。体验上，用户是无密码登录加一键支付；安全上，抗钓鱼、卡号不触达商户；合规上，商户的 PCI 范围明显缩小。

这里也要避免误解：Passkey 不是支付本身，Click to Pay 也不是登录本身。Passkey 管身份，Click to Pay 管卡凭证和支付网络。两者结合，才是“现代支付架构”的完整价值。

## 16. 五、CIT / MIT 与业务场景

CIT 是 Cardholder Initiated Transaction，也就是买家在场的交易。比如首笔支付、3DS、SCA，用户明确参与了这次支付。

MIT 是 Merchant Initiated Transaction，也就是商户发起的交易。比如订阅续费、免密支付、分期扣款，用户不一定在场。

订阅、一键支付、复购业务都建立在 CIT/MIT 的规则上。首次绑卡或首次授权必须是 CIT，通常还要做 3DS 或 SCA。这个动作的目的不是单笔交易本身，而是建立后续 MIT 的授权依据。

后续 MIT 必须带首次授权留下的凭证，比如 token、agreement ID 或 mandate。金额、频率、场景不能超出用户授权范围。

订阅业务最怕两件事：扣款失败重试没设计好，卡过期没有自动更新。Network Token 能解决一部分卡过期问题，因为卡到期或换卡时，卡组织可以自动更新 token，减少 involuntary churn。

## 17. 补充：3DS + CIT/MIT 时序图

这一页用时序图讲订阅和免密扣款的合规逻辑。

第一步是 CIT 首笔，用户在场，完成 3DS 或 SCA。这个动作不是只为了完成当前交易，更重要的是建立后续扣款的授权基础。

第二步是 PSP、3DS 服务和发卡行完成认证与授权，并返回授权结果。

第三步是商户保存 token、agreement ID 或 mandate。注意这里保存的是授权凭证，不是卡号，更不是 CVV。

第四步是后续 MIT。商户基于首次授权凭证发起订阅续费、免密扣款或分期扣款。

关键约束是：MIT 不是任意扣款。金额、频率、场景都不能超出用户首次授权范围，且后续请求要能关联到首次授权凭证。

## 18. 六、Tokenization

Tokenization 是支付产品落地里最核心的一页。

判断 Token 要看四个维度：能不能跨 PSP，是否可逆，泄露后的风险，绑定粒度是什么。

PSP token 最简单，成本低，但通常只能在这个 PSP 内使用。换 PSP 以后，原来的 token 可能失效。所以如果你只接一家 PSP，它很好用；如果你做多 PSP，它会成为限制。

Merchant token 一般通过 Vault 实现，可以跨 PSP，但意味着你要承担 Vault 的合规成本。它适合交易体量足够大、对路由和控制权要求高的团队。

Network Token 是被很多团队低估的能力。它由卡组织发放，不只是安全能力，也是业务增长能力。它可以提升授权率，卡过期时自动更新，并且通常绑定商户或设备，泄露以后也更难被滥用。

Payment method ID 更偏用户绑定层，方便业务识别用户的支付方式，但它不是跨 PSP 编排的完整答案。

多 PSP 场景的核心痛点就在这里：PSP A 的 token 到 PSP B 不能用，所以多 PSP 策略经常会倒逼你做独立 Vault，或者尽量使用 Network Token。

## 19. 补充：Network Token 生命周期

这一页补 Network Token 的生命周期。

第一步，商户或 PSP 向卡组织 token 服务请求 Network Token。第二步，卡组织发放 token，并根据规则绑定商户、设备或使用场景。

第三步，授权交易时使用 Network Token 替代 PAN。这样商户和部分链路不需要直接暴露真实卡号。

第四步，卡过期、换卡、补卡时，卡组织可以自动更新 token。这个能力对订阅业务很关键，因为它能减少因为卡过期导致的扣款失败和用户流失。

第五步，在多 PSP 场景下，Network Token 能缓解 PSP token 被某一家服务商锁定的问题。它不是所有场景都天然跨 PSP，但比单一 PSP token 更适合作为长期支付凭证体系。

这一页的关键结论是：Network Token 不只是安全能力，也是授权率、续费成功率和多 PSP 韧性的基础能力。

## 20. 案例三：Network Token 降低订阅扣款失败

这个案例用来说明 PCI 不只是成本。

场景是订阅业务里，很多续费失败来自卡过期、换卡、补卡以后原凭证失效。用户没有主动取消，但扣款失败，最后形成被动流失。

误判点是只把 token 当成脱敏工具，没有利用卡组织 token 的生命周期能力。

影响是续费失败率上升，客服和重试成本增加，多 PSP 路由也受 PSP token 锁定限制。

正确做法是优先评估 Network Token，结合账户更新、智能重试和多 PSP 路由。Network Token 可以在卡生命周期变化时自动更新，在部分场景下提升授权率和续费成功率。

这个案例的结论是：好的合规设计可以转化成授权率、续费率和容灾能力。

## 21. 模块三：技术架构

模块三给架构和工程同学。

如果业务确实需要更深地处理支付，比如多 PSP、订阅、风控、自建路由，那就绕不开 Vault、密钥、前端支付页安全和数据治理。

这部分的目标不是鼓励大家自建，而是把自建的成本和边界讲清楚。

## 22. 七、Card Vault 选型

Vault 选型主要看三件事：PSP 数量、交易量、团队能力。

如果只有一家 PSP，优先使用 PSP Vault，不要过度设计。PSP Vault 成本最低，Scope 最小，团队负担也最低。

如果有多家 PSP，又不想自己承担完整 PCI Vault 的建设成本，可以考虑第三方 Vault，比如 VGS、Spreedly、Basis Theory 这类服务。选之前一定要看对方的 AOC 和能力边界。

如果交易量足够大、路由策略足够复杂、团队有专门的安全和支付工程能力，才考虑自建 Vault。自建 Vault 的成本不只是开发成本，还包括 HSM/KMS、双人控制、密钥轮换、渗透测试、审计、权限、监控和事件响应。

独立 Vault 的真正价值在编排。比如按成本路由、按授权率路由、按地区路由、按卡类型路由，或者 PSP A 故障时快速切到 PSP B。

但要强调：没有足够交易量和团队能力，自建 Vault 很容易把业务收益变成长期负担。

## 23. 八、密钥管理

密钥管理的核心是分层和隔离。

典型模型是 DEK、KEK、Master Key。DEK 用来加密具体数据，比如 PAN；KEK 用来加密 DEK；Master Key 在 HSM 或 KMS 中保护 KEK。每一层只加密下一层。

信封加密的做法是：数据库里存密文和被 KEK 加密后的 DEK，KEK 保存在 KMS 或 HSM 里。这样即使数据库被拖库，攻击者拿到的也只是密文和被包住的 DEK，没有 KEK 仍然解不开。

常见翻车点有几个：密钥硬编码进代码，密钥进 Git，密钥和密文放同一个库，生产密钥复制到测试环境，用邮件或聊天工具发密钥。

PCI v4.0 对密钥轮换、弱算法淘汰、早期 TLS 禁用都有更明确要求。密钥生命周期也要完整管理：生成、分发、使用、存储、轮换、吊销、销毁，每一步都要有记录。

## 24. 九、前端支付页安全

前端支付页安全经常被忽略。很多团队会说“卡数据不走我的服务器，所以我安全”，但 Magecart 不看服务器，它直接在用户浏览器里偷卡。

Magecart 的典型方式是恶意 JavaScript 注入支付页，监听用户输入卡号，然后把卡号发到攻击者服务器。

英航 2018 年的案例很典型：恶意代码只有 22 行，运行了 22 天，导致 38 万卡号泄露，最后罚款金额非常高。这个案例说明，前端脚本治理不是可选项。

风险来源包括广告脚本、埋点 SDK、客服浮窗、A/B 测试工具、CDN 篡改。它们看起来都不是支付核心系统，但只要运行在支付页，就有能力读取页面环境。

控制手段包括 CSP、SRI、脚本白名单、变更审批和完整性监控。v4.0 6.4.3 明确要求支付页所有脚本必须有清单、有授权、有完整性监控。不是加一个 CSP 就结束，而是要能证明每个脚本为什么存在、谁批准、有没有被篡改。

## 25. 十、数据治理（日志 / 客服 / BI / AI 风控）

数据治理要问的问题是：卡数据除了 CDE 里那一份，其他副本在哪里。

日志系统里有没有 PAN？APM trace 里有没有请求体？客服系统有没有截图？BI 数仓有没有同步原始卡号？测试环境有没有真实卡？这些都是 Scope 扩散的来源。

日志原则是全链路不记录 PAN 和 CVV。必须做正则脱敏、字段级过滤、访问权限控制和留存周期管理。

客服系统只展示卡品牌和后四位，不展示完整卡号。客服操作要全量审计，权限要拆分，不能因为排查问题就临时给一堆人看敏感数据。

BI 和数仓里，PAN 应该替换成 token 或 hash，CVV 绝对不能进入，BIN 在很多风控和分析场景中可以保留，但也要明确用途和权限。

AI 风控会和数据最小化产生冲突。模型想要更多数据，PCI 要求更少数据。可行的平衡方式包括用 BIN 代替 PAN，用 token 作为用户键，用 PSP 风控分作为特征，用合成数据训练。

v4.0 强调数据发现分类、留存处置策略、访问权限定期复核。这些不只是文档，要能证明在持续执行。

## 26. 模块四：运营审计

最后一个模块讲运营审计。

前面讲的是怎么判断、怎么设计、怎么落地。最后还要回答一个问题：你怎么证明这些控制一直有效。

PCI v4.0 的方向很明确，不希望企业年底突击补材料，而是要求持续运营。研发测试、权限、审计证据、事件响应，都要进入日常流程。

## 27. 补充：审计证据地图

这一页讲审计证据地图。

PCI v4.0 的重点不是年底补材料，而是持续证明控制有效。审计要看的不只是你有没有制度，而是制度有没有被执行，执行有没有证据。

核心材料包括 SAQ、AOC、ROC、ASV 扫描和渗透测试。它们证明你做过正式验证。

运营证据包括变更记录、权限复核、安全测试修复闭环、事件响应演练记录。这些证明你的控制不是某一天有效，而是在日常运营里持续有效。

审计时最怕的是控制写在制度里，但没有工单、扫描结果、审批记录、复核记录作为证据。所以证据库要日常沉淀，最好和工单系统、CI/CD、扫描平台、权限系统打通。

## 28. 十一、研发与测试

研发和测试环节有两条红线。

第一，测试环境禁止使用真实卡。测试要用 PSP 提供的测试卡，或者使用脱敏仿真数据。真实卡进入测试环境，通常是审计里的严重问题。

第二，密钥不能进 Git。无论是代码仓库、CI 日志、配置文件，还是工单截图，只要密钥泄露，就要立即轮换，并排查是否需要重加密。

CI/CD 对支付代码要有更严格的变更控制。任何影响 CDE 的变更，都应该走正式流程，有审批、有测试、有回滚方案。

安全测试不能只做一次。SAST、DAST、SCA、ASV 扫描、渗透测试、支付页完整性检测，都要有结果跟踪和修复闭环。

v4.0 还强调开发、测试、生产隔离，安全测试结果必须跟踪修复，支付开发人员需要专项安全培训。这里的培训不是通用安全意识课，而是和支付、卡数据、CDE 相关的专项培训。

## 29. 十二、审计与事件响应

事件响应最重要的是时间线。

如果发生 PAN 泄露，通常需要在 24 小时内通知 PSP、收单行或卡组织，并启动取证、隔离和根因分析。错过通知窗口，后续罚金和处置成本可能上升。

如果 CVV 被记录，问题更严重。因为 PCI DSS 本身禁止存储 CVV，所以这不是普通泄露，而是控制设计已经违规。处理方式是立即删除、全面排查、做根因分析，并确认是否还有其他系统记录了 CVV。

如果密钥泄露，要立即轮换密钥，评估是否需要重加密历史数据，并按等同 PAN 泄露的严重程度处理。

如果支付页 JS 被篡改，要先下线或隔离页面，保留证据，清理恶意脚本，复查变更路径，然后加强 CSP、SRI 和完整性监控后再上线。

v4.0 要求事件响应计划每年至少演练一次，关键联系人 24/7 可达，复盘报告要保留。演练可以是桌面推演，也可以是实战演练，但不能只停留在纸面流程。

## 30. 培训主线 · 收尾

最后把今天的主线串起来。

PCI DSS 不只是把卡数据暴露面降到最小，它也可以转化成业务收益。

Network Token 可以提升授权率、降低卡过期导致的订阅流失。Token 编排可以支撑多 PSP 路由、成本优化和容灾。PayFac 可以支撑商户规模化覆盖。Click to Pay 和 Passkey 可以减少商户接触卡数据，同时提升用户体验。

四个模块的逻辑是：合规判断告诉我们要求是什么；产品落地告诉我们怎么通过设计降成本；技术架构告诉我们如果必须碰数据，怎么安全落地；运营审计告诉我们怎么持续证明做到了。

所以 PCI DSS 不应该是年底突击，也不只是安全团队的任务。它应该成为支付产品和架构设计的一部分，越早进入决策，成本越低，业务收益越明显。
